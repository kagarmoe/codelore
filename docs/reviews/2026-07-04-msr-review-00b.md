# MSR Expert Review — `00b-development-plan-evidence-first.md`

## 1. Top findings (by severity)

### F1. Window membership is undefined, and the Phase 2 exit criterion is circular — HIGH

**What:** The plan (Phase 2, exit criteria) requires that "commit and changed-file counts match the audit," but neither the plan nor the schema defines what "commits in the window" means. `git log v1.1.0..v1.2.0` (reachability set-difference) includes every commit merged since the last tag — including commits *authored months before the window* on long-lived branches merged late. First-parent traversal gives a different, smaller set (merge-level history). Tags placed on non-mainline commits (release branches, cherry-picked hotfixes) make the two definitions diverge sharply.

**Interpretation:** If the corpus audit (Phase 2 entry task 1) is produced by the same `rev-list` invocation the ingester will use, "counts match" verifies self-consistency, not correctness. This is exactly the kind of criterion that looks falsifiable but isn't.

**Recommendation:** Make window membership an explicit, documented policy in the audit memo: reachability definition (`end ^start`), traversal mode (all-ancestry, with first-parent annotated per commit), and which timestamp (author vs. committer date — they differ under rebase) is authoritative. Require the audit to be produced *independently* (e.g., GitHub compare view + manual tag inspection), so the match is a real cross-check. Record out-of-window author dates as a known artifact class, not noise.

### F2. `artifact_link` claims have no precision/recall requirement — HIGH

**What:** Phase 3a treats `artifact_link` as "hard-deterministic," and Phase 3b's audit is question-oriented (golden set), not link-oriented.

**Interpretation:** Merge-metadata links are deterministic; *reference-based* links ("explicit references," per the claim taxonomy) are regex extraction over shared issue/PR number space — a well-known precision hazard (`#123` may be an issue, a PR, a cross-repo ref, or literal text in a code block). The missing-link problem is one of the field's most replicated findings: Bird et al. showed bug-fix link samples are systematically biased (linked commits are not representative of all fixes), and recovery tools (ReLink and successors) exist precisely because explicit links are incomplete. Squash-merge workflows sever PR↔commit links entirely: the squash commit's SHA never appears in the PR's commit list; recovery depends on the `(#N)` message suffix or GitHub's commit→PR API endpoint. Deterministic ≠ correct, and deterministic ≠ complete.

**Recommendation:** Add to Phase 3a/3b exit criteria: (a) a hand-built ground-truth PR↔commit link set for the window (feasible at gastown scale), (b) reported link precision and a recall estimate against it, (c) `link_method` recorded per link (the schema's edge properties already provide this — use it in the pack, not just the graph). Treat "commit in window with no recoverable PR" as a counted, reportable category.

### F3. Tangled commits are not addressed anywhere — MEDIUM-HIGH

**What:** No phase, policy, or claim rule mentions commits that address multiple concerns.

**Interpretation:** Herzig & Zeller and subsequent replications found a substantial fraction (commonly cited near 15–20%) of change commits are tangled. A `change_fact` or `narrow_inference` claim scoped at commit granularity ("this commit guards behavior X") silently overclaims when the commit also contains a refactor and a docs fix. The one MVP inference rule (`behavior_guard_inference`: test change + code change in same PR) is *especially* exposed — co-presence in a tangled PR is weak co-change evidence.

**Recommendation:** Scope `change_fact` subjects at file/hunk granularity (the `diff_hunk` locator and `Evidence` span fields already support this); require `behavior_guard_inference` to cite the specific hunks, not the commit; add tangling to the warrant's `limitations` field by rule. Also handle the inverse: a changelog entry with no traceable artifact should be a named contradiction/abstention case — the evidence policy covers release notes *omitting* changes but not changelog entries *asserting* untraceable ones.

### F4. Golden question set methodology would not survive review — MEDIUM

**What:** Phase 2 entry task 2: hand-verified by (implicitly) one person; Phase 3b: "sampled claim audit … error rate recorded" with no sampling protocol.

**Interpretation:** MSR standards for manual labeling: documented sampling frame, stratification by claim type, at least two raters on a subsample with agreement reported (Cohen's kappa), a disagreement-resolution rule, and sample sizes that justify the reported error rate. Single-rater labeling is defensible for an MVP portfolio artifact only if declared as a threat to validity.

**Recommendation:** Specify n, stratification, and rater count in the Phase 3b exit criterion now; audit a sample of *abstentions* too (false abstention is the symmetric error the plan never measures); state the single-rater limitation explicitly in the Phase 7 packaging.

### F5. Phase 2/3 partially reinvents mature tooling — MEDIUM

**What:** Phase 2 git ingestion (tags, commits, diffs, changed files) and Phase 3b forge snapshotting.

**Interpretation:** PyDriller covers Phase 2's deliverables directly, including rename detection wrapping git's similarity heuristic. Perceval (GrimoireLab) is near-isomorphic to the snapshot/replay policy: it fetches GitHub items and preserves *full raw JSON with capture metadata and stable UUIDs* — the plan's reproducibility section, already built and battle-tested. Custom work is genuinely warranted for: window-membership policy (F1), span-level evidence, claims/warrants, stable ID scheme, and the validator. git2net is not relevant at this stage.

**Recommendation:** Adopt or at least study PyDriller/Perceval before writing extractors; if writing custom, document why in the Phase 3b entry decision. For GitHub snapshots, plan for: rate limits and pagination (the issues endpoint returns PRs — a classic double-count bug), deleted/ghost users, edited or deleted comments (a re-fetch that diverges from a snapshot should surface as a contradiction, not a crash), and force-pushed PR branches whose commits exist only via the API, not the clone. GHTorrent-era lesson: forge data is *mutable and deletable*; the snapshot-as-source-of-truth stance is the right defense — say so and pin the API version.

### F6. Symbol lineage (Phase 8) assumes tooling the plan never builds — MEDIUM (post-MVP)

**What:** Phase 8 promises module→symbol→concept lineage; canonicalization Tier 2 cites "same symbol lineage across refactors."

**Interpretation:** The ordering is correct and matches tooling maturity: file/module lineage is tractable (rename detection, with known failure modes — it's a pairwise similarity heuristic at diff time, threshold-sensitive, not stored in history, and `git log --follow` is single-file and unreliable through splits). Method-level history needs AST-diff tooling (GumTree-style; CodeShovel/CodeTracker for method tracking) — language-specific and imperfect. Concept lineage has no established tooling at all. Module-only in MVP is implementable as scoped.

**Recommendation:** Record the rename-detection threshold and tool version as provenance on module lineage decisions; treat symbol lineage as a research-grade sub-project when Phase 8 arrives, not an increment.

### F7. Author identity is a bare string — LOW (for MVP)

`Commit.author` in `models.py` has no aliasing/merging policy (identity merging is a known MSR problem; bot accounts inflate metadata). Harmless *only because* MVP makes no author-level claims. Note the gap so no future claim type quietly consumes the raw field.

## 2. What is strong

- **Snapshot/replay reproducibility** (Cross-Cutting Policies) directly answers the GHTorrent-era mutability problem; captured-at semantics and checksums are the right primitives.
- **Confidence reflects evidence quality, not model confidence** — precisely the discipline missing-link research says linking heuristics need. `link_method`/`membership_type` edge properties anticipate mixed-quality links.
- **The canonicalization policy** (tiers, counterevidence, `POSSIBLY_SAME_AS`, `REUSES_LABEL`, refusing name-matching) is better than most published identity-resolution practice; Module-only MVP scope is realistic.
- **Gating graph infrastructure on retrieval value** matches repeated MSR/industry experience that graph stores rarely beat structured filtering for single-window questions.
- **Corpus audit before extractor code** is exactly right — most MSR failures start with unverified corpus assumptions.

## 3. What is weak or risky

- Window membership semantics (F1) and link measurement (F2) are the two places where "deterministic" is doing rhetorical work the underlying git/GitHub semantics don't support.
- Tangling (F3) is a validity threat to the plan's *only* inference rule.
- llama.cpp prediction: gastown's release window will validate almost nothing that llama.cpp stresses. llama.cpp squash-merges to master with very frequent build tags; PR links survive mainly via `(#N)` message suffixes; monthly windows will contain thousands of commits. First breakage will be (a) evidence-pack and rendered-view scale, then (b) the golden-question methodology, which doesn't transfer to a window with no curated narrative. Phase 8's exit criteria don't yet name a scale bound.

## 4. Open questions

1. Which commit set defines the window — all-ancestry, first-parent, or both recorded with distinct membership types? (The schema's `NATIVE_TO_WINDOW`/`LINKED_TO_WINDOW` split could carry this — is that intended?)
2. Is gastown squash-merge, merge-commit, or mixed? The audit should report this; it determines whether `IMPLEMENTED_BY` is recoverable from git at all.
3. What is the target n and rater protocol for the Phase 3b claim audit?
4. Will forge snapshots be re-fetched ever, and if so, is snapshot divergence modeled as a `Contradiction`?
5. At what artifact count does the pack format or per-claim rendering stop being reviewable — and should that bound be an explicit Phase 8 exit criterion?
